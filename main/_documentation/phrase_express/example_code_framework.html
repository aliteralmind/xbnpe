XBNPE: Example code framework (XCF) project

An "XCF documentation file" (or "XCF-file") is a plain HTML file containing PhraseExpress example code. Each piece of example-code in an XCF-file, called an "XCF-example", is automatically created into an actually-usable ("executable") phrase--so that it may be called by your users. In addition, all the XCF-examples in a single XCF-file may be executed as a whole: one after the other, in order to ensure the example code is working as expected.

In all cases, an XCF-example snippet is written once, in a single location: The HTML documentation file. Whenever a change is made to an XCF-example, its XCF-file must be re-processed, in order to re-create its executable phrases.

XCF-example code:

  • [go]   As displayed in documentation
  • [go]   HTML source code
    • [go]   Open marker
    • [go]   Close marker
    • [go]   Line-ending non-breaking spaces
  • [go]   Executable phrases:
    • [go]   Plain executable
    • [go]   "wsc" executable

XCF-file:

  • [go]   Test all examples in an XCF-file

[top]   XCF-example: Display-version

XCF-example: [display, HTML source, executable phrases]

This is how an XCF-example looks in documentation:

dq1quotient
(*Input:*)
            {#SETPHRASE -description 
   zvDIVIDEND_TOP -content 
   65 
            -autotext zvDIVIDEND_TOP -folder ctvv_folder}{#SETPHRASE -description 
   zvDIVISOR_BTM -content 
   10 
            -autotext zvDIVISOR_BTM -folder ctvv_folder}

(*Go:*)
   {#insert zQuotient_M_divisorDividend}

(*Output:*)
   {#insert zvDIVIDEND_TOP}/{#insert zvDIVISOR_BTM}={#insert zvQUOTIENT}

(It divides 65 by 10, ignoring any remainder. Output is stored in zvQUOTIENT.) This example may be executed with either

  • dq1quotient
  • dq1quotientwsc

(This XCF-example is duplicated from

[top]   XCF-example: Executable

XCF-example: [display, HTML source, executable phrases]
Executable: [plain, wsc, all]

There are three types of executable phrases created by the
    xcfbuild
process. Two phrases allow each XCF-example to be individually executed, and one additional phrase allows all examples in an XCF-file to be executed together.

  • [go]   Plain executable: Execute the XCF-example, exactly as displayed in the documentation.
  • [go]   "wsc" executable: The same as the plain-executable, with additional setup and cleanup code.
  • [go]   Test all XCF-examples in an XCF-file: Calls each example's wsc-executable, one after the other. This may also perform some overall setup and cleanup operations, as needed by all examples. Running this "exe-all" phrase is to verify all examples in an XCF-file work as intended.

XCF-example: [display, HTML source, executable phrases]
Executable: [plain, wsc, all]

With the exception of spacing (indentation and empty lines), this phrase is no different than the display-version:

dq1quotient
(*Input:*){#SETPHRASE -description zvDIVIDEND_TOP -content 65 -autotext zvDIVIDEND_TOP -folder ctvv_folder}{#SETPHRASE -description zvDIVISOR_BTM -content 10 -autotext zvDIVISOR_BTM -folder ctvv_folder}(*Go:*){#insert zQuotient_M_divisorDividend}(*Output:*){#insert zvDIVIDEND_TOP}/{#insert zvDIVISOR_BTM}={#insert zvQUOTIENT}

(This is displayed in uncompressed format. Also see: import steps.)

The XCF-plain-executable phrase is intended to be called with all expected setup already completed. If necessary, this setup must be described before or after the XCF-example itself. In general, XCF-examples should be called from within a new document.

[BACK]   XCF-example: Executable: "wsc": With-setup-and-cleanup

XCF-example: [display, HTML source, executable phrases]
Executable: [plain, wsc, all]

The "with-setup-and-cleanup" ("wsc") executable phrase calls the plain-executable, with some setup and cleanup code before and after the call. For example:

dq1quotientwsc
dq1quotient:{#ENTER}{#insert dq1quotient}{#ENTER}{#insert zBalDoneSlk2}

Setup and cleanup code is defined in the XCF-example open and close markers.

[top]   XCF-example: HTML source-code

XCF-example: [display, HTML source, executable phrases]
HTML source: [open marker, line-ending non-breaking spaces, close marker]

Here is the HTML source code for the example XCF-example:

<PRE><!--xcfexample...START  //  dq1quotient:{#ENTER}--><B><U>dq1quotient</U></B>
(*Input:*)
            {#SETPHRASE -description&nbsp;
   zvDIVIDEND_TOP -content&nbsp;
   <B>65</B>&nbsp;
            -autotext zvDIVIDEND_TOP -folder ctvv_folder}{#SETPHRASE -description&nbsp;
   zvDIVISOR_BTM -content&nbsp;
   <B>10</B>&nbsp;
            -autotext zvDIVISOR_BTM -folder ctvv_folder}

(*Go:*)
   {#insert zQuotient_M_divisorDividend}

(*Output:*)
   {#insert zvDIVIDEND_TOP}/{#insert zvDIVISOR_BTM}={#insert zvQUOTIENT}<!--xcfexample...END  //  {#ENTER}{#insert zBalDoneSlk2}--></PRE>

XCF-specific portions are highlighted. Aside from the raw PhraseExpress code, there are three critical hidden elements that make up an XCF-example. The open marker:
    <!--xcfexample...START // ...-->
line-ending non-breaking spaces:
    &nbsp;
and close marker:
    <!--xcfexample...END // ...--></PRE>

Also note that indentation is done with three spaces. While tabs are acceptable, they result in more severe indentation.

The open and close markers contain set-up and clean-up code, that is only used in the "...wsc" executable phrase.

[BACK]   XCF-example: HTML source: Open marker

XCF-example: [display, HTML source, executable phrases]
HTML source: [open marker, line-ending non-breaking spaces, close marker]

All XCF-examples must be preceded by an open-marker:

<!--xcfexample...START  // ...-->

Any set-up code must be declared following the slash-slash-space ("// "). Although not required, it is strongly recommended that something always be done. At a minimum, consider notifying your users of the about-to-be-executed code. For examples that print output:
    <!--xcfexample...START // dq1quotient:{#ENTER}-->
For examples with no printed output:
    <!--xcfexample...START // {#BALLOON -title Please wait... -text Beginning nameofxcfexample}-->

[BACK]   XCF-example: HTML source: Line-ending non-breaking spaces

XCF-example: [display, HTML source, executable phrases]
HTML source: [open marker, line-ending non-breaking spaces, close marker]

            {#SETPHRASE -description&nbsp;
   zvDIVIDEND_TOP -content&nbsp;
   <B>65</B>&nbsp;

Line-ending non-breaking spaces are required to preserve proper spacing in PhraseExpress executable code that, when displayed, are separated by line-breaks. These non-breaking spaces also allow XCF-example-code to be selected and copied directly from within a web-browser. Once

  1. fully unindented, and
  2. reformatted (such that newlines are replaced with nothing),
the code will be equivalent to the executable phrase. (Note that the phrases name must be separated from the phrase-body.)
