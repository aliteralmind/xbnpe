XBNPE: Large Project Database Framework (LPD) project

The "LPD Framework" defines a "database" for a sufficiently-large PhraseExpress project. A project that uses the lpd-framework is called an "LPD Project". The lpd-framework (or simply, "lpd") defines a specific database "table", and "columns" within those tables--it has no knowledge of the actual data contained therein. An lpd-project's database is defined in an "LPD Configuration File".

The lpd-framework provides a large set of utilities for creating, reading, analyzing, duplicating, and debugging your project's data. Providing this basic data-management functionality, allows you to more quickly build the custom aspects of your project.

Global internal-variables     [top]

There are three critical internal-variable used by all lpd-projects. These variables are temporary, in the sense that they are only used during specific lpd-processes (such as 'kcttpmd'). Once the process is complete, all these variables (phrases) may be safely deleted:

zvPRJ_ABRV_1C

Holds the lpd project abbreviation.

zvDTYPE_ABRV_1C

Holds the data-type abbreviation.

zvCVT_FRMV_1C

Holds the post-from "_M_" abbreviation

When put together...

...they comprise the beginning of many lpd-related phrase names. For example:
    zz\\\{#insert zvPRJ_ABRV_1C\\\}\\\{#insert zvDTYPE_ABRV_1C\\\}
is the beginning of all configuration-variables in every lpd-project. Such as
    zzMnvCmd
in the MINV-Command project, or
    zzTpmd
in the example TextPad Macro Creator project.

    zz\\\{#insert zvPRJ_ABRV_1C\\\}\\\{#insert zvDTYPE_ABRV_1C\\\}\\\{#insert zvCVT_FRMV_1C\\\}

LPD-Framework: Basic definitions     [top]

A single "row" of data in your database "table" is called a "Data Instance". A data-instance is a single item in your project, and is made up of a well-defined set of phrases, called "Configuration Variables" (each config-var is a single "cell" in the row). Here is an example data instance.

Public and Private-keys

Each data-instance has a public-key and private-key. Unrelated to encryption, these are the unique identifiers for a data-instance. The "Public Key" is user-facing--it is seen and used by you and your users, and is the first piece of information needed whenever anything is done to or with a data-instance. The private key is generally short, and should always be somewhat memorable.

The "Private Key" distinguishes the phrase-names of one data-instance from another's, and is used for internal purposes only. Once configured, the private-key is never seen or used again, and should never be altered. The private-key is either equal to the public-key (when the public-key is very short), or and abbreviated version of it.

CV-Type, and Project and data-type abbbreviations

Every configuration-variable has a "CV-Type", which is essentially the name of a single "column" in the database-table. For example, here is a config-var in the MINV-Command project:
    zzMnvCmdEtrActionName1C
This phrase-name is in the format
    zz[project][data-type][key][cv-type]

  • [project]: The MINV-Command project is abbreviated "Mnv"
  • [data-type]: This particular project has two data-types: command, abbreviated "Cmd", and variable, abbreviated "Var". For projects with only a single data-type, it is possible for this to be the empty-string ('', or nothing).
  • [key]: This config var is for the "enter" MINV-Command. In other words, in the MINV-Command project, this is the "enter" data-instance. It's key is "Etr".
  • [cv-type]: The cv-type of this configuration-variable is "ActionName1C".

One additional and unrelated definition: When a string is in "One Case" form, its first character is uppercase, and all others are lowercase.

LPD-Framework: Multi-Typed LPD-Projects     [top]

Having an lpd-project with two or more data-types has significant implications.

Link between data-types

Each data-type in a "Multi-Typed LPD-Project" (such as the MINV-Command project, which has "commands" and the "variables" required by them) are configured in separate lpd-config-files. Just as two related database tables must be linked via one of its keys, so is the case here. There must be a "link" between a non-key cv-type in data-type A, and a key in data-type B (I use the public-key).

A MINV-Command requires exactly one MINV-Variable in order (for its master-phrase) to properly function. A Variable, however, may be used by multiple Commands. This is a one-to-many relationship, where the Variable is the one-side. This implies that the "non-key side" of this relationship is the Command data-type, and the "key-side" is the Variable.

The MINV-Command project has the non-key cv-type [VarAbrv1CForCmd], which is a link to the MINV-Variable's public key, [Abrv1C].

In a multi-typed lpd-project, loading a data-instance actually requires loading a single instance from all data-types. For example, the MINV-Command must be loaded, along with its required Variable. This is called a "Multi-Typed Data Instance".

There is no direct support for multiple data-types built into the lpd-framework. You need to create these "linking" functions in order to manage your multiple types.

This needs to be documented, but for now, see the actual MINV-Command link-phrases in xbn_phrase_express\phrase_express\build_minv\bmnv__link_cmds_and_vars.px

Recommendation: Use the same internal-variable name on all sides of the link

The MINV-Command's [VarAbrv1CForCmd] and MINV-Variable's [Abrv1C] cv-types, share the same internal-variable name. While this results in this value being redundantly-loaded (when the command-instance is loaded, and again when the variable-instance is loaded), using the same internal-variable name is recommended. Note that this refers only to the internal-variable name. The rest of the cv-type elements, such as the from-abrvs and config-var phrase-name postfixes should be different across all data-types.

LPD-Config-File: Global configuration     [back]

The first three lines in an LPD-config-file are global "inline" configuration-variables, and apply to all cv-types.

{#insert zzInlineCVOutDir}

Project metadata access phrase: [OutputFileDir]

The directory in which all output files are written.

Specific requirements: The first line in an LPD-config-file must begin with "{#insert zzInlineCVOutDir}=", and must be immediately followed by the full directory path. There may be no whitespace, the value is required (may not be empty-string), and the directory it refers to must exist.

{#insert zzInlineCVPrjAbrv1C}

Project metadata access phrase: [ProjectAbrv1C]

The short abbreviation for this LPD-project, which is used in many phrase-names in the output files. The post-zz abbreviation for the TextPad Macro Creator project (TPMC) is "Tpmd" (d for Demo, to distinguish it from my actual "Tpm" LPD-project), which results in phrase names for all TPMC config-variables beginning with 'zzTpmd'.

It should be alpha-numeric, with the first character uppercase, and the rest lowercase. This is used in a lot of phrase names, and should therefore be short--it is recommended to be no more than four characters.

{#insert zzInlineCVDataTypeAbrv1C}

Project metadata access phrase: [DataTypeAbrv1C]

The type of data-instance represented by the configuration-variables in this LPD-config-file. Along with the project-abbreviation, this is used in a lot of phrase names, and should therefore be short--no more than four characters. For example, in the MINV-Command project, there are two data-types

  • commands, abbreviated "Cmd", and
  • variables, abbreviated "Var"

When your project only has one data-type, then its abbreviation may be set to the empty-string ('', or nothing). Doing so is recommended.

For projects with more than one data-type, each type must contain at least one character, and must be unique among all data-types in the project. In addition, the first character in each data-type must also be unique, since it is used in some project-utility shortcuts (for example, 'kcttpmd', which creates a blank configuration template for the MINV-Command project). This is not programatically-enforced--you must manually ensure this uniqueness.

The private key cv-type block must be preceded by a comment that begins:

Line 2: Post-from ("_M_") abbreviation     [back]

  • Example value: Pnv1
  • CV-Type metadata access phrase: ...cvtFromAbrv1C

The "Post-From Abbreviation" (post "_M_"), which is called the "From Abrv", serves two purposes. From the point of view of your lpd-project's users, this is the text that follows "_M_" (or "_st") in phrase names (such as
    zDoSomething_M_pnv1
) when a particular internal-variable must be set, in order for the phrase to function properly. This is a naming convention used throughout XBN-PhraseExpress.

For lpd-project developers only, the from-abrv also is the unique identifier for most of the phrase names of a cv-type's metadata. For example, take the "phrase-name-abbreviation" cv-type (abbreviated "Pnv1") in the example TextPad Macro Creator project. Here are all of its metadata phrases:

  • zzTpmdPnv1_cfgValue_M_key
  • zzTpmdPnv1_cfgVarPNmPost
  • zzTpmdPnv1_intrnlVarNm
  • zzTpmdPnv1_cfgVarPhrsName_M_key
  • zzTpmdPnv1_helpForDlg
  • zzTpmdPnv1_valuePrintable
  • zzTpmdPnv1_validateValuePhrsNm
  • zzTpmdPnv1_nkcvListNum
  • zzTpmdPnv1_keyCVTFromAbrv1C
The only phrase whose phrase-name is not identified by the from-abrv, is the phrase whose value is the from-abrv:
    zzTpmdPhrsNmAbrv1C_cvtFromAbrv1C
This name of this one phrase is uniquely identified by the phrase-name postfix ("PhrsNmAbrv1C"), as discussed in the next section. Conversely, the value of the phrase-name-postfix is stored in a phrase whose name is uniquely identified by the from-abrv:
    zzTpmdPnv1_cfgVarPNmPost

Where a data-instance has a private-key, the cv-type metadata has the post-from abbreviation.

Line 3: Configuration-variable phrase-name postfix     [back]

  • Example value: PhrsNmAbrv1C
  • CV-Type metadata access phrase: ...cfgVarPNmPost

Like the from-abrv, the cv-type's "Phrase-Name Postfix" serves two purposes. For your users, this is the postfix of all configuration-variables of this type. For example, take these two data-instances in the example TextPad Macro Creator project. Here are the private-key configuration-variables for these two macros:

  • "go-to-prev-x-mark": zTpmdGtpxPhrsNmAbrv1C
  • "go-to-next-x-mark": zTpmdGtnxPhrsNmAbrv1C

For lpd-project developers only, the phrase-name postfix also as the unique portion of the single cv-type's utilty that stores the value of the from-abrv. For example:
    zzTpmdPhrsNmAbrv1C_cvtFromAbrv1C
whose value is "Prv1"

Conversely, the phrase-name postfix itself is stored in a phrase whose name is uniquely identified by the from-abrv:
    zzTpmdPnv1_cfgVarPNmPost

Where a data-instance has a public-key, the cv-type metadata has the phrase-name postfix.

Line 4: Internal (temporary) variable name     [back]

  • Example value: zvPHRS_NM_ABRV_1C
  • CV-Type metadata access phrase: ...intrnlVarNm

Must be unique among all internal-variables in the project, including those internal variables in all other data-types within the same project (note the exception for links in multi-typed lpd-projects).

Line 5: Validate value phrase-name     [back]

  • Example values:
    • {#insert zzValidateValueUnwantedFlag}
    • zVldtPhrsNmAbrv1C_M_self
  • CV-Type metadata access phrase: ...validateValuePhrsNm

The name of the phrase that can validate values of this cv-type. When no validation is wanted, set this to
    {#insert zzValidateValueUnwantedFlag}
Otherwise, set this to the name of the phrase that will do the validation. It is strongly recommended that the phrase end with "_M_self" (at least "self"), indicating that the value must be set into its internal-variable.

This phrase is never called by the lpd-framework. It is merely stored.

This may require other cv-types as well, depending on the order in which you plan to read in the configuration-variables. It is up to you to ensure all variables are set as required in the from-clause ("_M_...").

  • Example value: The unique~identifier that distinguishes the phrase~names of the configuration~variables for each TextPad macro. Recommended to be an abbreviated version of the public~key.

LPD-Config-File: All non-key cv-types     [back]

All non-key cv-types ("nkcv") exist immediately after the public-key. This section must begin with this exact comment (no extra text allowed):
    (*{#insert zzRqdCmtStartAllNKVarsBelow}*)

After this is followed by each cv-type block. For example:

1. (**)
2. Kc
3. KeyCommand
4. zvKEY_COMMAND
5. {#insert zzLpdValueNotPrintable}
6. {#insert zzValidateValueUnwantedFlag}
7. The key~command, as configured in TextPad > Preferences > Keyboard > Macros. Must be QUINTUPLY=escaped==as in '{#insert zOYb}#CTRL {#insert zOYb}#HOME{#insert zCY2b}'. If the macro takes a particularly long time to execute, then add some {#insert zOYz}#sleep...{#insert zCYz} after triggering the key command.

An nkcv-block has the same restrictions as the private- and public-keys. The only differences are in lines 1 and the addition of line 5.

Line 1: Potentially-empty comment

While this comment is required, it may be empty. As with the private and public key, it may contain additional text, as long as the line immediately preceeding line two ends with "*)", and there is no other '*)' between it and the comment-start.

Line 5: Printable flag

  • The only possible values:
    • zzLpdValuePrintable, which equals {#insert zzLpdValuePrintable}, or
    • zzLpdValueNotPrintable, which equals {#insert zzLpdValueNotPrintable}
  • CV-Type metadata access phrase: ...valuePrintable

Is the value of configuration-variables of this type "printable"? If it can be safely duplicated (meaning the duplicated value is exactly the same as the original), then it is "printable". If it contains any phrase-calls, then it cannot be safely duplicated.

"Loading" a data-instance "into memory"

An lpd-project can "load" a single data-instance "into memory", by storing its values into its "temporary internal variables". As configured in the lpd-config-file, each cv-type has a specific internal-variable associated to it. Here are all internal-variables in the example TextPad Macro Creator project:

CV-Type Internal variable name
[PhrsNmAbrv1C] zvPHRS_NM_ABRV_1C
[MacroAbrv1C] zvMACRO_ABRV_1C
[LivePNPostTC] zvLIVE_PN_POST_TC
[KeyCommand] zvKEY_COMMAND
[Steps] zvMACRO_STEPS

These variables are "temporary" because they are treated as such. They are only used during the life of a process. Once that process is complete, all may be safely deleted. Temporary-internal-variables are an XBN-PhraseExpress convention, and are named beginning with "zv", and are followed with all-uppercase words, divided by underscores.

Loading a data-instance into memory, versus directly reading config-vars

The lpd-project utility zMnvCmdSetAllIVarsToCfgValsWPubkPrompt loads an entire data-instance into memory, making all of the above internal-variables are available. Depending on the number of cv-types in your project, and the size of each variable's value, this may take a few seconds. For one-time-only processes, this is usually sufficent.

For processes that happen frequently or need to be fast, loading a data-instance into memory is likely unacceptable. In this case, configuration-variables need to be directly accessed, by first setting the data-instance's keys (along with its project and data-type abbreviations) with (for example)
    zTpmdPromptSetPubSetPrivKeys
which prompts you for the macro's public-key, and then accessing each config-var with
    ..._cfgValue_M_key
For example, the macro's "LivePNPostTC" (used when other phrases want to execute a macro) is accessed with
    zzTpmdPnv1_cfgValue_M_key

Internally, the public-key is always stored as "one case" ("1C"), meaning the first character is uppercase, and all others are lower. For example, here is the internal-variable of the public-key as configured for the MINV-Command project:
    zvACTION_NM_1C

"plpdcf": The central LPD-Framework process     [top]

"plpdcf" ("Process the LPD Config-File") is the central process in the lpd-framework--it essentially creates your database, based on the configuration in the lpd-config-file. The result of the "plpdcf" process is a directory full of plain text files:

  • metadata and utilities for each cv-type (one file per-cv-type)
  • metadata for the project as a whole (one file)
  • utilities explicitely tailored to the project (one file)
Each line in these text-files is an individual phrase. Once all files are imported into PhraseExpress, they may be used as specified throughout this documentation.

Sub functions and utilities:

  • "blanklpdcf": Prints out an empty lpd-config-file.
  • "plpdcfm": Writes project-metadata file only
  • "plpdcfmu": Writes project-metadata and utility files
  • "plpdcfps": Writes cv-type utility files only
  • "plpdcfu": Writes project-utility files only

This process takes some time. On my computer, it takes an average of thirty seconds per cv-type to complete the entire process. However, this is a one-time process. The only time it needs to be run again, is when a change is made to the lpd-config-file.

Known Issues

  • The project-utilities file is incorrectly based on project metadata that has already been imported into PhraseExpress. -- Workaround: Generate the project-metadata, import it, then generate the project-utilities.
  • A mis-formatted line in any cv-type's config-block will cause the entire process to fail. -- Workaround: Run the metadata-only sub-process, "plpdcfm", which reads (much more) quickly through all cv-types, and crashes at the first error found. Once the metadata-file starts printing, quit the process (with the ESC-key) at any point before the file is saved.
